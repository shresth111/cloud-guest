# Location Domain -- Database Schema (Smart Location Provisioning additions)

Covers the schema additions Smart Location Provisioning made on top of
Module 006's original `locations` table (see `LOCATION_ARCHITECTURE.md` for
that original design). Migration: `alembic/versions
/0026_add_location_provisioning.py`. See `FLOW.md` for the full design
reasoning behind each of these.

## `locations` (extended)

| Column          | Type          | Nullable | Notes                                                                                     |
|-----------------|---------------|----------|---------------------------------------------------------------------------------------------|
| `property_type` | `VARCHAR(30)` | Yes      | `app.domains.location.enums.PropertyType` value (e.g. `hotel`, `cafe`, `office`, `custom`). Every pre-existing row has none; optional even on the plain create endpoint. |
| `location_code` | `VARCHAR(30)` | Yes      | Auto-generated by `LocationService.create_location` for *every* newly-created location (e.g. `LOC-2026-000001`). Nullable at the column level only so this migration never has to backfill a value for a pre-existing row. |

Indexes added: `ix_locations_property_type` (plain btree), and
`uq_locations_location_code` -- a **partial** unique index
(`WHERE location_code IS NOT NULL`) so uniqueness is enforced only among
rows that do have a code, mirroring `organization_members`'s own
partial-unique-index precedent (migration `0003`).

## `location_code_counters` (new table)

A structural mirror of BE-013 Part 4's `invoice_number_counters`
(`app.domains.billing.models.InvoiceNumberCounter`):

| Column        | Type          | Nullable | Notes                                                        |
|---------------|---------------|----------|---------------------------------------------------------------|
| `counter_key` | `VARCHAR(50)` | No       | `"location:<year>"` -- one row per calendar year.             |
| `last_value`  | `INTEGER`     | No       | Default `0`; incremented atomically (see below).             |

Plus the standard `BaseModel` columns (`id`, `created_at`, `updated_at`,
`deleted_at`, `is_deleted`, `created_by`, `updated_by`, `version`) and a
unique index on `counter_key`.

Incremented via exactly one atomic Postgres statement per call --
`INSERT ... ON CONFLICT (counter_key) DO UPDATE SET last_value =
last_value + 1 ... RETURNING last_value` (see
`app.domains.location.number_generator`'s module docstring for the full
concurrency-safety write-up) -- never a racy `SELECT MAX(...) + 1`.

## `users` (extended)

| Column                  | Type      | Nullable | Notes                                                                                          |
|--------------------------|-----------|----------|--------------------------------------------------------------------------------------------------|
| `must_change_password`  | `BOOLEAN` | No       | `server_default false` (every pre-existing row becomes `false`). Set `true` by Smart Location Provisioning for a freshly-provisioned owner account; cleared back to `false` by `AuthService.change_password`/`reset_password`. See `FLOW.md` §9. |

## Not migrated

* `app.domains.billing.constants.PlanFeatureKey`'s new members
  (`DASHBOARD` ... `MULTI_LOCATION`, `MAX_CONCURRENT_SESSIONS`,
  `MAX_STAFF_USERS`, `MAX_API_KEYS`) -- `PlanFeature.feature_key` is a
  plain `String` column, never a native Postgres enum type, so adding
  `StrEnum` members is a code-only change.
* `app.domains.rbac.enums.AuditAction`'s new members
  (`LOCATION_PROVISIONED`, `LOCATION_WELCOME_EMAIL_SENT`) -- same reason;
  `AuditLogEntry.action` is a plain `String`.
* No new RBAC permission group/action/scope row -- `locations.manage` was
  already seeded since BE-004.
