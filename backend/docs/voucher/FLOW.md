# Voucher: Flows and Design Decisions

This document records every design decision this module made where the
brief left room for judgment, plus the end-to-end batch-lifecycle/
redemption flow. Read this before modifying `app/domains/voucher/`.

## 1. Plaintext code storage, not hashed

**Decision: `Voucher.code` is stored as a plain, unique `String` column --
never hashed.**

Every other short-lived, randomly-generated secret in this codebase is
hashed at rest: `app.domains.otp.models.OtpRequest.code_hash`,
`app.domains.router.models.RouterProvisioningToken.token_hash`,
`app.domains.router_agent.models.RouterAgentCredential.credential_hash`.
A voucher code deliberately does not follow that precedent:

* An OTP code / provisioning token / agent credential is generated by the
  platform and delivered electronically directly to the party who will
  present it back -- nothing else ever needs to *read* it again. A voucher
  code is, by this module's whole purpose, a physical or verbally-
  communicated artifact: printed on a card, written on a receipt, read
  aloud at a front desk. The platform must be able to display and export
  it in plaintext after generation (`GET .../export`, `GET .../vouchers`)
  -- there is no way to "export a hash" and have it be useful to a print
  vendor or a receptionist. Hashing would defeat this module's core
  function.
* The threat models differ. An OTP code's value is that it proves the
  *current* holder of a phone/email received it moments ago; a stolen hash
  is useless without also compromising the delivery channel's timing
  window. A voucher code's value is "does this string exist and is it
  still valid" -- identical whether read from a plaintext column or
  reconstructed from a hash comparison. Hashing buys no additional
  protection against the actual risk here (a guest finding/guessing a
  valid code), which is instead mitigated by generation entropy
  (`constants.VOUCHER_CODE_ALPHABET` + `VoucherBatch.code_length`) and
  redemption rate limiting (`VoucherRedemptionRateLimiter`), not by
  secrecy of the stored row.

This is a documented judgment call, not carelessness: `code` still carries
a real database `UNIQUE` constraint, and every lookup goes through
SQLAlchemy's parameterized queries -- injection safety was never the
reason other domains hash their secrets, and plaintext storage here does
not weaken it.

## 2. Approval workflow shape

**Decision: `VoucherBatchStatus` has the full five-state graph the module
brief asked for (`DRAFT -> PENDING_APPROVAL -> APPROVED -> ACTIVE ->
EXPIRED/REVOKED`), but this module's required API surface exposes no
dedicated "submit" or "activate" endpoint -- `create_batch` performs the
submission, and `approve_batch` performs the activation, each within the
same HTTP call that reaches it.**

The required endpoint list (`POST /voucher-batches`, `.../approve`,
`.../revoke`, ...) has no "submit for approval" or "activate" endpoint.
Rather than leave `DRAFT`/`APPROVED` as states only reachable by direct
repository manipulation (never exercised through any real flow), this
module folds the adjacent transitions into the calls that already exist:

* `POST /voucher-batches` creates the row (`DRAFT`), then immediately calls
  `VoucherService._submit_batch` (`DRAFT -> PENDING_APPROVAL`) -- so every
  batch's audit trail always shows a real, distinct
  `voucher_batch_submitted` event, even though the caller only ever made
  one HTTP call.
* `POST /voucher-batches/{id}/approve` calls
  `VoucherService._approve_and_activate`, which performs **both**
  `-> APPROVED` (recording `approved_by_user_id`/`approved_at`, auditing
  `VOUCHER_BATCH_APPROVED`) **and** `APPROVED -> ACTIVE` (auditing
  `VOUCHER_BATCH_ACTIVATED`) as two separate, independently-validated
  transitions against `constants.VOUCHER_BATCH_STATUS_TRANSITIONS`, one
  right after the other. There is no operational scenario in this module's
  own scope where an approved-but-not-yet-active batch needs to sit idle
  before going live -- unlike `app.domains.router`, where a physical device
  must check in before `PROVISIONING -> ONLINE` can ever fire, a paper/
  print vendor is not waiting on a distinct "turn it on" signal separate
  from "approve it".

The full graph is real and exercised (both edges are independently
validated and independently audited), just not independently reachable
over HTTP -- a future decision to add a dedicated submit/activate endpoint
needs no schema or graph change, only a new route calling the existing
`_submit_batch`/`_approve_and_activate` methods directly.

### The `voucher.manage` fast path

**Decision: a caller who holds `voucher.manage` may create a batch that
skips the `PENDING_APPROVAL` queue entirely.**

`app.domains.rbac.seed.expand_grant_level` expands `GrantLevel.OPERATE` to
every applicable action *except* `MANAGE`/`DELETE` -- so `APPROVE` is
already included in `OPERATE` (Location Manager/Reception Staff/Guest
Operator system roles all hold `VOUCHER: OPERATE`), while only `FULL`
includes `MANAGE`. A caller who holds `voucher.manage` is therefore
already strictly more privileged than one who can merely approve -- letting
them skip the queue for their own newly-created batch is a small,
reasonable convenience, not a privilege escalation (they could always
create-then-immediately-approve-themselves as two calls; this makes it
one). `router.py`'s `dependencies.get_voucher_manage_bypass` determines
this via a non-raising `AccessValidator.has_permission` check -- distinct
from the mandatory `voucher.create` `RequirePermission` gate the route
itself carries -- and passes the boolean result into
`VoucherService.create_batch` as `has_manage_permission`. When `True`,
`create_batch` routes the freshly-submitted batch straight through
`_approve_and_activate` (the exact same method the `approve_batch` endpoint
calls), so `approved_by_user_id`/`approved_at` are always populated
identically regardless of which path reached `ACTIVE`.

This mirrors BE-009's own "create + approve both required, but a
sufficiently-privileged actor can do both" precedent for router enrollment
(`RouterProvisioningService.submit_enrollment`/`approve_enrollment`),
adapted to a single-call rather than two-call shape since vouchers have no
physical device to wait on either.

## 3. Bulk code generation

**Decision: `secrets.choice` over a from-scratch, print-friendly alphabet
(`constants.VOUCHER_CODE_ALPHABET = "ABCDEFGHJKLMNPQRSTUVWXYZ23456789"`),
with in-memory + database collision retry, and outright rejection (not
chunking) of a quantity exceeding `MAX_BULK_CREATE_SIZE`.**

* **Alphabet.** Uppercase letters + digits, excluding every character a
  person could misread off a printed card or misdictate over a phone call:
  `0`/`O`, `1`/`I` (both upper and lowercase are avoided entirely -- the
  whole alphabet is uppercase-only, so there is no lowercase `l` to confuse
  with anything either). 22 letters (26 minus `I`/`O`) + 8 digits (10 minus
  `0`/`1`) = 30 symbols. This is the same design intent as Crockford's
  Base32 alphabet, independently derived here (Crockford's own alphabet
  also folds a checksum scheme this module has no use for).
* **Collision retry.** `VoucherService._generate_codes` generates
  candidates in-memory (deduplicating within the same batch for free, via
  a `set`), then checks the *database* for any of those candidates already
  existing (`VoucherRepository.find_existing_codes`, a single `WHERE code
  IN (...)` query) and discards any hits, repeating for up to
  `constants.CODE_GENERATION_MAX_ROUNDS` (10) rounds. A database-level
  collision is exceedingly rare at any sane `code_length` (30^8 possible
  8-character codes), but is still handled for real rather than assumed
  away -- `VoucherCodeGenerationExhaustedError` is the defensive backstop
  if 10 rounds still can't fill the requested quantity (e.g. an
  unreasonably short `code_length` combined with a large `quantity`).
* **Bulk-size handling: reject, don't chunk.** A requested `quantity`
  greater than `app.database.constants.MAX_BULK_CREATE_SIZE` (1,000) is
  rejected outright (`VoucherBatchQuantityExceededError`) rather than
  silently split across multiple `bulk_create` calls. Chunking would defeat
  the atomicity a single `bulk_create`'s single flush already provides --
  a mid-way constraint violation on chunk 3 of 5 would leave a batch with
  a partial, silently-wrong voucher count with no natural place to detect
  or roll it back. A caller needing more than 1,000 vouchers creates
  multiple batches -- a small UX cost for a real correctness guarantee.
  The same `MAX_BULK_CREATE_SIZE` bound is reused (not redefined) for
  `POST /vouchers/import`'s code list.

## 4. `expires_at`: set at first redemption, not generation

**Decision: `Voucher.expires_at` starts `NULL` and is computed as
`redeemed_at + batch.validity_minutes` on the *first* successful
redemption -- never recomputed on subsequent redemptions of the same
multi-use voucher.**

`VoucherBatch.validity_minutes` documents "how long a voucher grants
access **once redeemed**" -- it is a duration relative to first use, not a
wall-clock deadline fixed at generation time. A voucher sitting unredeemed
in a drawer for three weeks has not "used up" any of its post-redemption
validity window; only `VoucherBatch.batch_expires_at` (a separate,
generation-time shelf-life -- see §5) can invalidate an unredeemed code.
Computing `expires_at = generated_at + validity_minutes` at generation
time would silently expire vouchers nobody had a chance to use yet, which
is backwards from this module's own stated semantics -- this is explicitly
called out because it is easy to get backwards. For a multi-use voucher
(`max_uses_per_voucher > 1`), the *second* and later redemptions check the
`expires_at` set by the *first* redemption and never push it forward --
"3 uses within 24 hours of first use", not "3 uses, each extending the
clock by another 24 hours".

## 5. Two distinct expiry dimensions

**Decision: batch-level shelf-life (`batch_expires_at`) and per-voucher
post-redemption validity (`validity_minutes`/`expires_at`) are two
separate mechanisms, never conflated -- mirroring OTP's own "two distinct
rate-limit dimensions" discipline applied here to expiry instead.**

1. **Batch shelf-life** (`VoucherBatch.batch_expires_at`, nullable) -- a
   generation-time deadline: codes not redeemed by this timestamp become
   permanently invalid, independent of how long a voucher would grant
   access once used. Checked lazily on every read
   (`VoucherService._refresh_batch_expiry`): an `ACTIVE` batch past this
   timestamp is flipped to `EXPIRED` in the same call, no background
   sweep/cron needed -- mirrors `app.domains.otp.models.OtpRequest
   .is_expired`'s identical "checked on read" posture. Once a batch reads
   as anything other than `ACTIVE`, redemption/validation against any of
   its vouchers raises `VoucherBatchNotActiveError` -- see that
   exception's docstring for why this, not a dedicated expiry error, is the
   right shape here (a batch not being active covers "still pending
   approval", "revoked", and "expired" uniformly; the caller does not need
   a different exception *type* to know a code currently isn't usable,
   only the human-readable reason, which the response's `batch_status`
   data field carries).
2. **Post-redemption validity** (`VoucherBatch.validity_minutes` /
   `Voucher.expires_at`) -- see §4. Checked per-voucher at
   validate/redeem time; a voucher past its own `expires_at` raises
   `VoucherExpiredError` specifically (distinct from
   `VoucherBatchNotActiveError`, since the batch itself may still be
   perfectly `ACTIVE` -- it is this one voucher's own clock that ran out).

## 6. Validation vs. redemption, and distinct exceptions

**Decision: `validate_voucher` is a pure read-only check; `redeem_voucher`
performs the actual state mutation. Both share one internal
`_redemption_failure_reason` classifier, but raise the same set of
distinct exceptions per reason -- never one collapsed "invalid voucher"
error.**

Mirrors `app.domains.otp.exceptions`'s identical reasoning: a voucher code,
like an OTP identifier, is not a persistent account being enumerated
(anyone holding the code already knows it's a real code), so a distinct
error per failure reason is better guest-facing UX ("this code has already
been fully used" vs. "this code has expired") with no meaningful new attack
surface. What must never leak: another valid code, or a redemption count
beyond what the caller's own request already implies -- neither exception
here echoes either.

`redeem_voucher`'s state machine, once a voucher passes every check:

* **First use** (`status == UNUSED`): sets `redeemed_at`,
  `redeemed_identifier`, `expires_at` (see §4), `use_count = 1`, and
  transitions to `EXHAUSTED` (if `max_uses_per_voucher == 1`, the default)
  or `ACTIVE` (otherwise).
* **Subsequent use** (`status == ACTIVE`): increments `use_count`/
  `last_used_at` only -- `expires_at`/`redeemed_identifier` are untouched.
  Transitions to `EXHAUSTED` once `use_count` reaches
  `max_uses_per_voucher`.

## 7. Guest-facing endpoints: no RBAC, but rate limited

**Decision: `POST /vouchers/validate`/`POST /vouchers/redeem` carry no
`RequirePermission`/`CurrentUser` dependency at all.**

Mirrors `app.domains.otp.router`'s identical justification (see that
module's `FLOW.md` §5): the caller is a guest at a captive portal, by
definition with no platform-user identity or JWT to present -- there is no
RBAC permission a guest could ever be granted. Abuse protection comes
entirely from `VoucherRedemptionRateLimiter`, a Redis-backed,
per-`source` (the caller's presumed IP address, resolved by
`dependencies.get_redemption_source`) INCR+EXPIRE+TTL throttle -- the
identical mechanism shape `app.domains.otp.service.OtpRateLimiter`
establishes, applied to a different Redis key
(`constants.VOUCHER_REDEMPTION_RATE_LIMIT_KEY_TEMPLATE`). Scoped by source
rather than by presented code: the risk being defended against is one
source trying many codes, not one code being tried by many sources (a
front-desk device validating a stack of printed vouchers in quick
succession is a real, non-adversarial pattern that must not be choked by a
per-code limit). The default threshold
(`DEFAULT_REDEMPTION_MAX_ATTEMPTS_PER_WINDOW = 30` per
`DEFAULT_REDEMPTION_WINDOW_MINUTES = 1`) is deliberately more generous than
OTP's own request limiter (5/hour): guessing a voucher code is a
single-shot action with no OTP-style two-step round trip to protect.

Both endpoints still use the standard `ApiResponse` envelope (unlike
`app.domains.router_agent`/`app.domains.wireguard`'s device-facing
endpoints) -- their real caller is the captive-portal *frontend*, a real
web/app client that benefits from the same structured contract every other
user-facing endpoint returns, the identical reasoning OTP's own guest-
facing endpoints already established.

## 8. CSV export / pre-printed code import

**Decision: `GET /voucher-batches/{id}/export` returns raw `text/csv`, not
`ApiResponse`-wrapped JSON -- the one deliberate deviation from this
codebase's "always `ApiResponse`" convention.**

A CSV file a print vendor or front-desk spreadsheet opens directly cannot
usefully be JSON-string-wrapped -- doing so would force every consumer to
first unwrap an envelope before the content becomes a real CSV file,
defeating the endpoint's entire purpose (a downloadable artifact).
`VoucherService.export_batch_csv` builds the CSV in-memory with the
stdlib `csv` module (`csv.writer` + `io.StringIO`) -- no new dependency --
and `router.py` returns it via a plain Starlette `Response` with
`media_type="text/csv"` and a `Content-Disposition: attachment` header.

**Decision: `POST /vouchers/import` means "bulk-import a batch of
pre-printed codes from an external system/print vendor", implemented for
real, with partial-success reporting.**

Vouchers are normally platform-generated, so "import" needed a concrete,
coherent meaning rather than being skipped. The meaning chosen: an
organization that already had codes printed by an external vendor (or
migrated from a legacy system) can register those exact codes against an
existing `VoucherBatch` (which may have `quantity: 0` -- see
`schemas.VoucherBatchCreate.quantity`'s docs -- meaning "no
platform-generated codes, only imported ones", or may already hold
generated codes and simply be adding more). Each submitted code is
normalized (stripped, upper-cased) and checked for uniqueness both within
the request itself and against every code ever generated/imported for any
batch (`VoucherRepository.find_existing_codes`, the same query
`_generate_codes`'s own collision check uses). **Partial success**, not
all-or-nothing: a real print run commonly has a handful of genuinely-
duplicate codes among an otherwise-good batch, so valid codes are inserted
and rejected codes are reported back individually with a reason
(`"duplicate within import request"` / `"code already exists"` /
`"empty code"`) rather than the whole call failing because of a handful of
bad rows. Importing into a batch that is `EXPIRED`/`REVOKED` is rejected
outright (`VoucherBatchNotActiveError`) -- there is no reason to add
codes to a batch that can never go live again.

## 9. Audit-volume judgment call -- and why vouchers differ from OTP

**Decision: batch lifecycle events (created/submitted/approved/activated/
revoked) and pre-printed code imports are audited, same as every other
domain's own lifecycle events. Unlike OTP, `redeem_voucher` audits *every*
successful redemption, not just a curated subset of failures.**

OTP's own audit-volume judgment call (`app.domains.otp.service`'s module
docstring) does **not** audit the routine "request a code" event at all --
reasoning that it is a high-volume, unauthenticated action that grants
nothing by itself (it is only step one of a two-step login). This module
makes a genuinely different call for its own analogous "value-transfer"
moment: **every successful `redeem_voucher` call writes one
`audit_log_entries` row (`AuditAction.VOUCHER_REDEEMED`)**. Unlike an OTP
*request*, a voucher redemption **is** the moment real network access is
granted -- often standing in for a real monetary/access transaction a
business may be legally or operationally required to have an audit trail
for ("who used voucher code X, when, from what identifier"). This is
consistent with, not a departure from, OTP's own precedent that a genuine
*success*/value-transfer event (`OTP_VERIFIED`) is audited; the difference
is that a voucher's redemption *is* that success moment, where OTP's
*request* is not.

What is **not** audited: read-only `validate_voucher` calls (no state
change, and a captive portal may legitimately poll this before committing
to a redemption) and *routine* (non-adversarial) redemption failures --
mirroring OTP's own tiering exactly, only `revoked`/`exhausted` (an
attempted reuse of a voucher already known to be dead -- adversarially
relevant) are audited (`AuditAction.VOUCHER_REDEMPTION_FAILED`);
`not_found`/`batch_not_active`/`expired` are routine guest-side churn (a
guest found an old, dead code, or tried before the batch went live),
logged via the structured logger but never written to the audit table.

## 10. `organization_id`/`location_id`: real FKs, one non-nullable

**Decision: unlike OTP's nullable `organization_id`/`location_id`,
`VoucherBatch.organization_id` is a real, **non-nullable** FK; only
`location_id` is nullable.**

A voucher batch is always created by/for a specific tenant -- there is no
"platform-wide, no-organization" voucher batch the way an OTP request might
plausibly carry no tenant context at all (e.g. an early captive-portal
integration test). `location_id` remains nullable because a batch may
legitimately be org-wide (usable across every location) or scoped to one
specific location. Both are validated for real at creation time (unlike
OTP, which explicitly does not validate its own nullable FKs) --
`VoucherService.create_batch` composes with the real
`OrganizationService`/`LocationService` (via narrow
`OrganizationLookupProtocol`/`LocationLookupProtocol` shapes) to confirm
the organization exists, and that a supplied location actually belongs to
it (`CrossOrganizationLocationAccessError` otherwise) -- the identical
composition-over-duplication precedent
`app.domains.router_provisioning.service.RouterProvisioningService`
already establishes for its own `LocationLookupProtocol` use.
`VoucherService.get_batch`/every other batch-scoped method additionally
enforces `CrossOrganizationVoucherBatchAccessError` when a caller's
resolved `requesting_organization_id` (from `X-Organization-Id`) doesn't
match the batch's own `organization_id` -- mirroring
`app.domains.router.exceptions.CrossOrganizationRouterAccessError`'s
identical tenant-boundary check.

## End-to-End Flow

1. **Create.** `POST /voucher-batches` (requires `voucher.create`).
   `VoucherService.create_batch`:
   a. Validates `quantity` (`<= MAX_BULK_CREATE_SIZE`) and `code_length`
      (`[MIN_CODE_LENGTH, MAX_CODE_LENGTH]`).
   b. Confirms the organization exists and (if a location was given) that
      it belongs to that organization -- raises
      `CrossOrganizationVoucherBatchAccessError`/
      `CrossOrganizationLocationAccessError` otherwise.
   c. Inserts the `VoucherBatch` row (`DRAFT`), audits
      `VOUCHER_BATCH_CREATED`.
   d. If `quantity > 0`, bulk-generates that many unique codes (§3) and
      bulk-inserts `Voucher` rows (`UNUSED`).
   e. Auto-submits (`DRAFT -> PENDING_APPROVAL`, §2).
   f. If the caller holds `voucher.manage`, immediately
      `_approve_and_activate`s the batch (§2).
2. **Approve.** `POST /voucher-batches/{id}/approve` (requires
   `voucher.approve`) -- `_approve_and_activate`: `-> APPROVED` (records
   `approved_by_user_id`/`approved_at`, audits `VOUCHER_BATCH_APPROVED`),
   then `APPROVED -> ACTIVE` (audits `VOUCHER_BATCH_ACTIVATED`).
3. **Revoke.** `POST /voucher-batches/{id}/revoke` (requires
   `voucher.update`) -- any non-terminal status `-> REVOKED`, cascading to
   every non-terminal `Voucher` in the batch (a single bulk `UPDATE`, not
   an ORM loop -- see `repository.VoucherRepository
   .bulk_revoke_vouchers_for_batch`), audits `VOUCHER_BATCH_REVOKED`.
4. **Validate.** `POST /vouchers/validate` (guest-facing, no RBAC) --
   rate-limits by source, looks up the code, and raises the first
   applicable of: `VoucherNotFoundError`, `VoucherRevokedError`,
   `VoucherExhaustedError`, `VoucherBatchNotActiveError`,
   `VoucherExpiredError` -- or returns a read-only view (`uses_remaining`,
   `is_first_use`, `expires_at`) with no mutation.
5. **Redeem.** `POST /vouchers/redeem` (guest-facing, no RBAC) -- the same
   checks as validate, then mutates per §6, and (on success) audits
   `VOUCHER_REDEEMED` (§9).
6. **Export/Import/Stats.** `GET .../export` (§8, `voucher.export`),
   `POST /vouchers/import` (§8, `voucher.import`), `GET .../stats`
   (`voucher.read`, a single grouped-count query --
   `VoucherRepository.get_batch_status_counts` -- yielding
   total/unused/active/exhausted/expired/revoked + a computed
   `redemption_rate`).
